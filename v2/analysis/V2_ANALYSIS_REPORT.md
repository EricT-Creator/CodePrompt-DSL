## 三、深度分析

### 3.1 F1: Compact Header (C) vs NL Baseline (A)

**方法**：对每个模型，逐任务配对比较 A 和 C 的 Pass@1，计算一致率（concordance）。

| 模型 | 层级 | n | A Pass@1 | C Pass@1 | Δ(C−A) | 一致率 |
|------|------|---|----------|----------|--------|--------|
| Claude-Opus-4.6 | Pro | 12 | 100% | 100% | 0% | 100% |
| GPT-5.4 | Pro | 12 | 100% | 100% | 0% | 100% |
| Gemini-3.0-Pro | Pro | 8 | 100% | 100% | 0% | 100% |
| GLM-5.0-Turbo | Mid | 12 | 100% | 100% | 0% | 100% |
| GLM-5.0 | Mid | 12 | 100% | 100% | 0% | 100% |
| Gemini-3.0-Flash | Mid | 12 | 100% | 100% | 0% | 100% |
| Kimi-K2.5 | Mid | 12 | 100% | 100% | 0% | 100% |
| MiniMax-M2.7 | Mid | 12 | 100% | 100% | 0% | 100% |
| Claude-Haiku-4.5 | 争议 | 12 | 91.7% | 100% | **+8.3%** | 91.7% |
| Gemini-3.1-flash-lite | Lite | 8 | 100% | 100% | 0% | 100% |
| DeepSeek-V3.2 | Lite | 12 | 83.3% | 91.7% | **+8.3%** | 91.7% |

**结论 F1**：

> **Compact Header (C组) 在约束遵循上与 NL Baseline (A组) 完全等效，甚至在边缘案例中略优。**

证据：
1. **9/11 模型** A 与 C 的配对一致率为 100%（完全相同的 Pass@1）
2. 唯二有差异的模型（Claude-Haiku-4.5 和 DeepSeek-V3.2），C 组反而**高于** A 组
   - Haiku: A=91.7% → C=100%（FE-04A 的 Tailwind 泄露在 C 组未出现）
   - DeepSeek: A=83.3% → C=91.7%（A 组缺 PY-02，C 组缺 FE-04，净效应 C 更好）
3. **没有任何模型出现 "C 低于 A" 的情况**

这是对 RQ1 的直接回答：紧凑 Compact Header **不会降低约束遵循成功率**。

### 3.2 F2: JSON (B) vs Compact (C)

**方法**：逐模型配对比较 B 和 C 的 Pass@1。

| 模型 | n | B Pass@1 | C Pass@1 | Δ(C−B) | 说明 |
|------|---|----------|----------|--------|------|
| Claude-Opus-4.6 | 12 | 100% | 100% | 0% | 等效 |
| GPT-5.4 | 12 | 100% | 100% | 0% | 等效 |
| Gemini-3.0-Pro | 8 | 100% | 100% | 0% | 等效 |
| GLM-5.0-Turbo | 12 | 100% | 100% | 0% | 等效 |
| GLM-5.0 | 12 | 100% | 100% | 0% | 等效 |
| Gemini-3.0-Flash | 12 | 100% | 100% | 0% | 等效 |
| Kimi-K2.5 | 12 | 100% | 100% | 0% | 等效 |
| MiniMax-M2.7 | 12 | 100% | 100% | 0% | 等效 |
| Claude-Haiku-4.5 | 12 | 100% | 100% | 0% | 等效 |
| Gemini-3.1-flash-lite | 8 | 100% | 100% | 0% | 等效 |
| **DeepSeek-V3.2** | **12** | **58.3%** | **91.7%** | **+33.3%** | **C 显著优于 B** |

**结论 F2**：

> **Compact Header (C) 与 JSON Spec (B) 在强模型上完全等效，但在弱模型（DeepSeek-V3.2）上 Compact 显著优于 JSON。**

证据：
1. **10/11 模型** B 与 C 完全等效（均为 100%）
2. DeepSeek-V3.2 是唯一的例外：B 组仅 58.3%（7/12，缺失 6 个文件），C 组 91.7%（11/12，缺失 1 个）
3. 差异的原因是 **DeepSeek-V3.2 在解析 JSON spec 格式的 prompt 时生成管线更易中断**——这不是"理解错误"，而是"生成失败"（文件不存在）

这个发现对论文有价值：它说明 **Compact Header 比 JSON spec 在弱模型上有更好的鲁棒性**——JSON 格式的冗余引号/括号可能干扰了 DeepSeek 的生成稳定性。

#### 3.3.3 敏感度分层

基于上述数据，模型可分为三个层次：

| 敏感度层 | 模型 | 特征 |
|---------|------|------|
| **Tier S：全域敏感** | Claude-Opus-4.6, GPT-5.4, Gemini-3.0-Flash, DeepSeek-V3.2 | 所有域的 A/B/C/D 变体均产生差异化输出 |

> **注**：Gemini-3.0-Pro 和 Gemini-3.1-flash-lite 缺少 FE 审查数据，仅基于 BE/PY 判定属于 Tier S。

**结论 F3**：

> **模型对 prompt 变体的敏感度存在明显的分层现象，且 BE/PY 域的变体敏感度系统性低于 FE 域。**

证据：
1. **7/9 个有完整 FE 数据的模型**在 FE 域展现了变体独立性（FLAG=0），说明模型**有能力**对不同 prompt 编码产生差异化响应
2. **同一批模型**（GLM-5.0、Kimi-K2.5 等）在 BE/PY 域却产出完全克隆的变体，说明**任务域本身影响了模型对 prompt 变体的敏感度**
3. 可能的解释：FE 任务（React 组件）的需求描述更丰富、代码结构更复杂，给模型留出了更多"因 prompt 不同而产生不同实现"的空间；而 BE/PY 基础任务（健康检查、CSV 解析等）的实现方式非常固定，模型倾向于输出"标准答案"

## 五、关键发现汇总

### 发现 1：Compact Header 与 NL 基线等效（强证据）

| 证据项 | 数据 |
|--------|------|
| Pass@1 配对一致率 | 9/11 模型 = 100%，2/11 模型 C > A |
| 没有任何 C < A 的案例 | 0 个模型 |
| CCR 对比 | A=0.990, C=0.992（C 略高） |

**结论强度**：强。Compact Header 在约束遵循层面与自然语言完全等效。

### 发现 2：Compact Header 优于 JSON Spec（弱-中证据）

| 证据项 | 数据 |
|--------|------|
| 10/11 模型 B=C | 天花板效应 |
| DeepSeek-V3.2: C >> B | 91.7% vs 58.3%（+33.3%） |
| B 组功能分系统性偏低 | BE 域 B=2.18 vs C=2.43 |

**结论强度**：中等。优势主要体现在弱模型上，强模型无差异。

### 发现 3：变体敏感度的模型-域交互效应（强证据）

| 证据项 | 数据 |
|--------|------|
| FE 域：8/9 模型独立 | 仅 MiniMax 全克隆 |
| BE 域：5/11 模型克隆 | GLM-5.0, Kimi, Haiku, GLM-Turbo(轻微), MiniMax |
| PY 域：4/11 模型克隆 | GLM-5.0, Kimi, Haiku, MiniMax |

**结论强度**：强。变体敏感度同时受模型能力和任务域影响。FE 域是研究 prompt 编码效应的最佳场景。

### 发现 4：Pass@1 天花板效应（事实）

| 证据项 | 数据 |
|--------|------|
| 整体 Pass@1 | 98.2% |

**影响**：v2 全量实验需要增加任务难度（进阶级任务），否则 Pass@1 无法作为有效的主指标。

### 发现 5：FE-04 是最具区分度的单一任务

| 证据项 | 数据 |
|--------|------|
| FE-04 整体 Pass@1 | 88.9%（唯一 < 95% 的任务） |

## 八、Token 成本分析

> **数据来源**：从 `v2/prompts/` 目录中 48 份 prompt 文件直接测量字符数（12 任务 × 4 变体）。

### 8.1 四组 Prompt 字符数（12 任务平均值）

| 变体 | 描述 | 平均字符数 | 相对 A 组 |
|------|------|-----------|----------|
| **A (NL)** | 自然语言基线 | **324** | — |
| **B (JSON)** | JSON Spec | **391** | **+20.7%**（更长） |
| **C (Compact)** | 英文紧凑 Header | **179** | **−44.6%**（节省） |
| **D (Classical)** | 古文 Header | **109** | **−66.3%**（节省） |

### 8.2 准确度 vs 成本矩阵

| 变体 | Pass@1 | CCR | 字符成本 | 结论 |
|------|--------|-----|---------|------|
| A (NL) | 97.6% | 0.990 | 324 字符 | 基线 |
| B (JSON) | 96.0% | 0.960 | 391 字符（**+21%**） | ❌ 更贵且更差 |
| C (Compact) | 99.2% | 0.992 | 179 字符（**−45%**） | ✅ **更便宜且等效/略优** |
| D (Classical) | 100% | 1.000 | 109 字符（**−66%**） | ⚠️ 字符最省，但需考虑 BPE |

### 8.3 逐任务 A vs C 的字符节省

| 任务 | A 字符 | C 字符 | 节省 | Pass@1 差异 |
|------|--------|--------|------|-----------|
| FE-01 | 306 | 139 | −54.6% | C ≥ A |
| FE-02 | 289 | 122 | −57.8% | C = A |
| FE-03 | 299 | 132 | −55.9% | C = A |
| FE-04 | 279 | 118 | −57.7% | C ≥ A |
| BE-01 | 291 | 156 | −46.4% | C = A |
| BE-02 | 343 | 214 | −37.6% | C = A |
| BE-03 | 310 | 179 | −42.3% | C = A |
| BE-04 | 320 | 189 | −40.9% | C = A |
| PY-01 | 322 | 186 | −42.2% | C = A |
| PY-02 | 384 | 248 | −35.4% | C > A |
| PY-03 | 394 | 258 | −34.5% | C = A |
| PY-04 | 346 | 210 | −39.3% | C = A |

**关键结论**：Compact Header (C 组) 在 12 个任务上实现了平均 44.6% 的 prompt 字符节省，且没有任何一个任务出现 C 的 Pass@1 低于 A 的情况。

### 8.4 重要说明：字符数 vs Token 数

上述数据使用的是字符数（char count），不是 token 数。这是因为不同模型使用不同的 tokenizer。但字符数和 token 数在英文内容上高度相关（约 1 token ≈ 4 chars）。**对于 D 组（古文），字符数的节省会被 BPE 对中文的低效编码部分抵消**——v1 已发现古文仅节省 4.6% token 而非 66% 字符。

对于 C 组（英文 Compact），字符到 token 的转换比相对稳定，预期实际 token 节省在 **35-45%** 区间。

### 8.5 B 组（JSON）为什么最差

| 方面 | B vs A |
|------|--------|
| 字符成本 | +20.7%（JSON 的引号、括号、冒号是冗余字符） |
| Pass@1 | −1.6%（96.0% vs 97.6%，因 DeepSeek 解析失败） |
| 功能分 | −0.12（2.65 vs 2.77） |

JSON 是最差的选择：比自然语言更贵，比 Compact Header 更长，且在弱模型上稳定性更差。

## 十、Pilot 数据局限性与后续实验规划

### 10.1 Pass@1 天花板效应

当前 12 个 pilot 任务的 Pass@1 达到 98.2%，9/11 模型满分。这意味着：
- Pass@1 在 Pro/Mid 层无区分力
- 差异几乎全部由 DeepSeek-V3.2 的文件缺失驱动
- **需要增加进阶难度任务以打破天花板**

### 10.2 变体克隆问题

BE/PY 域的大量变体克隆（5/11 模型全克隆）压缩了组间差异：
- 可能原因 A：任务太简单，模型收敛到"标准答案"
- 可能原因 B：平台缓存导致同一会话内的不同 prompt 返回相同结果
- **需要独立的克隆排除实验来区分这两种原因**

### 10.3 后续实验计划

基于 pilot 分析，确定两个独立的 follow-up 实验：

| 实验 | 目的 | 核心设计思路 |
|------|------|-------------|
| **EXP-A：进阶难度实验** | 打破天花板效应，提高 Pass@1 和功能分的区分度 | 增加反直觉约束 + 多步逻辑 + 约束冲突检测 |
| **EXP-B：克隆排除实验** | 区分"任务太简单"vs"平台缓存"vs"模型行为" | Seed 干扰 + 时间间隔重跑 + 进阶任务克隆对比 |

详细实验设计见：
