# V2 深度分析报告：六个机制性发现

> **分析时间**：2026-04-01 13:00 CST  
> **数据来源**：Pilot 528 条审计记录 + EXP-A 288 条审计记录 + EXP-B 综合数据  
> **分析立场**：不追求"哪个最好"，追求"差异的机制是什么"

## 发现 1：DeepSeek 的问题是"生成链路不稳定"，不是"能力不够"

### 1.1 数据支撑

把 DeepSeek-V3.2 的 Pass@1 拆成两阶段：

| 指标 | 数值 | 说明 |
|------|------|------|
| 总 Pass@1 | 40/48 = **83.3%** | 表面看起来"最差" |
| 文件生成率（Generation Success） | 41/48 = **85.4%** | 7 个文件直接没产出 |
| 条件 Pass@1（文件存在→约束通过） | 40/41 = **97.6%** | 一旦产出，几乎全过 |
| 条件平均功能分 | **3.00**（仅算已生成文件） | 和 Claude-Opus-4.6 并列最高 |

### 1.2 缺失文件的分布

| 变体 | 缺失数 | 缺失任务 |
|------|--------|---------|
| A (NL) | 1 | PY-02 |
| B (JSON) | **5** | FE-04, BE-01, BE-02, BE-03, BE-04 |
| C (Compact) | 1 | FE-04 |
| D (Classical) | 0 | 无 |

B 变体（JSON）集中缺失 5 个文件，说明 **JSON 格式的冗余括号/引号对 DeepSeek 的生成管线有系统性干扰**。

### 1.3 论文表述建议

**不应写**：
> "DeepSeek-V3.2 在代码生成能力上显著弱于其他模型"

**应写**：
> "DeepSeek-V3.2 展现了一种'能力-稳定性分离'现象：其条件解决方案质量（Conditional Solution Quality）与 Pro 层模型持平（R10=3.00），但生成成功率（Generation Success Rate）显著偏低（85.4% vs 其他模型 100%），且失败集中在 JSON 格式变体（B 组缺失 5/7），提示 Lite 层模型的生成管线对 prompt 格式的鲁棒性不足。"

### 1.4 这为什么重要

这把模型评估拆成了两个独立维度：

```
Total Performance = Generation Success Rate × Conditional Quality
```

在传统的 Pass@1 评估中，这两个维度被混为一体。但它们的改进路径完全不同：
- 生成成功率 → 工程侧优化（重试策略、格式容错、管线超时处理）
- 条件质量 → 模型能力侧优化（训练数据、指令遵循微调）

## 发现 2：D 组的"最强"是鲁棒性优势，不是语义优势

### 2.1 表面数据（容易误读）

| 变体 | Pass@1 | 看起来 |
|------|--------|--------|
| A (NL) | 97.6% | — |
| B (JSON) | 96.0% | 最差 |
| C (Compact) | 99.2% | 第二 |
| **D (Classical)** | **100%** | **"最好"** |

### 2.2 拆开条件质量后

只看**已成功生成的文件**的功能分均值：

| 变体 | 条件功能分均值 | 排名 |
|------|-------------|------|
| A (NL) | **2.77** | 1st |
| C (Compact) | **2.74** | 2nd |
| D (Classical) | **2.73** | 3rd |
| B (JSON) | **2.65** | 4th |

D 组的功能分实际上是四组中**第三**，而非最高。

### 2.3 D 组 100% Pass@1 的真正原因

D 组 100% 的唯一原因是：**DeepSeek-V3.2 在 D 变体上没有缺失任何文件**。

| 模型 | A 缺失 | B 缺失 | C 缺失 | D 缺失 |
|------|--------|--------|--------|--------|
| DeepSeek-V3.2 | 1 | **5** | 1 | **0** |
| 其他 10 模型 | 0 | 0 | 0 | 0 |

去掉 DeepSeek 后，A/B/C/D 四组在其余 10 个模型上的 Pass@1 **全部是 100%**。

### 2.4 正确的表述

**不应写**：
> "古文编码（D 组）取得了最高的准确率"

**应写**：
> "D 组（Classical）的 100% Pass@1 主要源于其在 Lite 层模型上更高的生成成功率——DeepSeek-V3.2 在 D 变体上的文件生成率为 100%（vs A=91.7%, B=58.3%, C=91.7%）。但在条件功能分上，D 组（2.73）低于 A 组（2.77），尤其在非中文模型上出现了功能分下降（GPT-5.4: D=2.50 vs A=2.67; Gemini-3.0-Flash: D=2.33 vs A=2.92）。D 组的优势更准确地描述为'传输层鲁棒性'——古文 Header 的极短 prompt 减少了弱模型生成管线中断的概率，而非在语义理解上带来了增益。"

## 发现 3：真正有区分度的不是 Prompt 形式，而是"生成稳定性 × 反直觉约束"

### 3.1 Pilot 中产生差异的来源分解

Pilot 的 528 条数据中，Pass@1 ≠ 100% 的数据点仅 9 个。逐个归因：

| 数据点 | 模型 | 任务 | 变体 | 失败原因 | 归因类别 |
|--------|------|------|------|---------|---------|
| 1 | DeepSeek-V3.2 | PY-02 | A | 文件缺失 | 生成中断 |
| 2-6 | DeepSeek-V3.2 | FE-04, BE-01~04 | B | 文件缺失 | 生成中断 × JSON格式 |
| 7 | DeepSeek-V3.2 | FE-04 | C | 文件缺失 | 生成中断 |
| 8 | Claude-Haiku-4.5 | FE-04 | A | R7 Tailwind 泄露 | 反直觉约束（plain_css） |
| 9 | Gemini-3.0-Flash | FE-04 | B/C? | R10=0（功能缺失） | 反直觉约束（plain_css） |

**归因统计**：
- **生成管线中断**：7/9 = 78%（全部来自 DeepSeek-V3.2）
- **反直觉约束违反**：2/9 = 22%（全部来自 FE-04 的 `plain_css`）
- **Prompt 编码理解错误**：**0/9 = 0%**

### 3.2 FE-04 为什么是唯一的区分任务

| 任务 | 约束特征 | Pass@1 | 是否反直觉 |
|------|---------|--------|----------|
| FE-01 登录表单 | tailwind + React + 无外库 | 100% | 否，模型默认套路 |
| FE-02 待办列表 | tailwind + React + 无外库 | 100% | 否 |
| FE-03 用户卡片 | tailwind + React + 无外库 | 100% | 否 |
| **FE-04 计数器** | **plain_css** + React + 无外库 | **88.9%** | **是，冲突了 tailwind 默认先验** |

FE-04 的 `plain_css` 约束违反了模型的训练先验（React 项目默认用 Tailwind），迫使模型主动抑制默认行为。这才是产生差异的**机制**。

### 3.3 EXP-A 验证了这个机制

EXP-A 的 12 个进阶任务全部包含反直觉约束，结果：
- 整体 Pass@1 = 97.6%（vs Pilot 98.2%），天花板略有松动
- **R10 功能分区间从 2.50-3.00 扩展到 4.12-4.98**（5 分制），区分度大幅提升
- Fail 原因从"生成中断"扩展到了"空文件"（Claude 3 个空文件），说明反直觉约束开始触发了更多类型的失败

### 3.4 论文级结论

> **Prompt 编码形式（NL / JSON / Compact / Classical）在当前 Pilot 任务集上不是代码生成差异的显著来源。** 真正产生可观测差异的是两个正交因素的交互：
> 
> 1. **生成管线稳定性**（model-level）：弱模型在复杂 prompt 格式（如 JSON）下更易中断
> 2. **约束-先验冲突度**（task-level）：当显式约束违反模型的训练先验时，差异才显性化
> 
> 这意味着 compact header 的研究价值不在于"哪种编码最好"，而在于**编码形式如何与模型的生成鲁棒性和约束覆写能力交互**。

## 发现 4：Backend 是真正的语义分层域

### 4.1 三域功能分对比

| 域 | 平均功能分 | 标准差 | 低分记录（≤1） |
|-----|----------|--------|-------------|
| Frontend | 2.89 | 低 | 极少 |
| **Backend** | **2.38** | **高** | **集中出现** |
| Python | 2.94 | 极低 | 几乎没有 |

Backend 的功能分不仅最低，而且**方差最大**——这说明它是暴露模型语义理解短板的最佳域。

### 4.2 Backend 任务的逐任务功能分模式

从 `func_variant_domain` 数据看 Backend 各变体的功能分：

| 变体 | Backend 功能分 | Frontend 功能分 | 差值 |
|------|-------------|---------------|------|
| A (NL) | 2.45 | 3.00 | **−0.55** |
| B (JSON) | **2.18** | 2.89 | **−0.71** |
| C (Compact) | 2.43 | 2.83 | −0.40 |
| D (Classical) | 2.43 | 2.83 | −0.40 |

B 组在 Backend 的功能分（2.18）是全数据集的**最低单元格**。JSON 格式在 Backend 域的表现系统性最差。

### 4.3 跨模型共性失败轮廓

多个模型在 Backend 上呈现同一种失败模式：

| 模型 | R10 分布 | 低分任务 |
|------|---------|---------|
| GPT-5.4 | 4个1分 + 13个2分 + 31个3分 | BE 域的 1 分集中在 R10 功能缺失 |
| GLM-5.0-Turbo | 4个1分 + 8个2分 + 36个3分 | 同上模式 |
| Kimi-K2.5 | 4个1分 + 8个2分 + 36个3分 | 同上模式 |
| Gemini-3.0-Flash | 1个0分 + 1个1分 + 10个2分 + 36个3分 | BE 域 2 分居多 |

**共性**：功能分为 1 的记录高度集中在 BE 域。BE 任务要求的 FastAPI endpoint 语义完整性（正确的状态码、边界条件处理、响应结构）比 FE 的 DOM 结构和 PY 的函数签名更难通过简单模板满足。

### 4.4 论文级结论

> **Backend API 域是检验模型语义实现能力的最有效域。** Frontend 和 Python 域的功能分接近天花板（2.89 和 2.94），区分力有限。而 Backend 域的功能分（2.38）存在明显的跨模型共性失败模式——多个模型在 BE-02（CRUD）、BE-03（注册验证）、BE-04（计算器边界）上系统性丢分，说明这些任务暴露的不是"某个模型特别弱"，而是"LLM 在 API 语义完整性上的共性短板"。
>
> 建议在论文中将 Backend 域的分析独立成小节，专门讨论 API 约束遵循的失败模式。

## 发现 5：克隆是"低自由度任务下的解坍缩"，不是"看不见 Prompt"

### 5.1 关键证据链

| 证据 | 来源 | 说明 |
|------|------|------|
| FE 域 8/9 模型独立 | Pilot R12 | 同一模型在 FE 域能区分变体 |
| BE/PY 域 5/11 模型克隆 | Pilot R12 | 同一模型在 BE/PY 域不区分变体 |
| 进阶任务零克隆 | EXP-A R12 | 增加复杂度后，连 Kimi 都不再克隆 |
| B3 矛盾测试 4/4 读了 Header | EXP-B B3 | 克隆模型（Kimi/GLM）在矛盾场景下主动提到了 Header 内容 |
| B1 Seed 测试 Kimi A=B=C | EXP-B B1 | 加 seed 后英文变体仍克隆，排除缓存 |

### 5.2 两种解释的证据权重

| 解释 | 支持证据 | 反对证据 | 判定 |
|------|---------|---------|------|
| **H-ignore: 模型忽略 Header** | 无 | B3 证明模型读了 Header | ❌ 排除 |
| **H-collapse: 低自由度解坍缩** | Pilot FE独立+BE克隆；EXP-A 进阶零克隆；B1 seed 无效 | 无 | ✅ 成立 |

### 5.3 "解坍缩"的机制解释

对于 BE-01 HealthCheck 这样的任务，合理的实现**只有一种模式**：

```python
from fastapi import FastAPI
from datetime import datetime
app = FastAPI()

@app.get("/health")
def health():
    return {"status": "ok", "timestamp": datetime.now().isoformat()}
```

无论 prompt 是 NL、JSON、Compact 还是 Classical，模型在生成时的"搜索空间"都坍缩到这个唯一解上。变量名可能不同（`health_check` vs `health` vs `get_health`），但结构同一。

这和 FE-01 LoginForm 形成对比——LoginForm 的合理实现有多种（state 管理方式、验证逻辑位置、错误提示 UI），因此模型有空间对不同 prompt 编码做出不同的实现选择。

### 5.4 后续研究价值

这个发现意味着 **"是否克隆"本身可以作为任务实现自由度的代理指标**：

```
Clone Rate(task) ≈ 1 / Implementation Degrees of Freedom(task)
```

如果能定量化这个关系，可以发展成一个独立的贡献点——用 prompt 变体克隆率来度量"任务的开放程度"。

### 6.3 论文叙事转向

**转向**：
> "在约束遵循型小任务上，Lite 层模型的条件解决方案质量已经与 Pro 层持平。Pro 层的优势不体现在'能不能做出来'，而体现在三个增量维度：更高的生成链路稳定性、更完整的 API 语义细节（尤其在 Backend 域）、以及更强的反默认先验约束处理能力。对于工程实践而言，这意味着**约束简单且格式标准的任务可以放心使用 Lite/Mid 模型**，将 Pro 模型的预算保留给需要复杂约束推理的场景。"

## 发现 7：三类模型行为模式——比能力层级更有解释力的分类框架

### 7.1 动机

前六个发现中反复出现一个现象：同一个模型在不同条件下表现出截然不同的行为（如 GLM-5.0 在 FE 域独立但 BE/PY 域克隆）。传统的 Pro/Mid/Lite 能力层级分类无法解释这种"同一模型内的分裂"。基于 Pilot + EXP-A + EXP-B 的全量数据，我们提出一种**行为机制导向**的三类分类。

### 7.2 三类定义

| 类型 | 核心特征 | 对 Compact Header 的意义 |
|------|---------|------------------------|
| **指令鲁棒型 (Instruction-Robust)** | 无论 header 格式如何，约束都执行；全域产出独立变体 | Header 可靠作为硬约束使用 |
| **格式敏感型 (Format-Sensitive)** | header 的结构化程度影响生成成功率/约束遵循率 | 需要选择合适的 header 格式 |
| **内容收敛型 (Content-Convergent)** | 任务有"标准答案"时直接克隆，不响应 header 变化 | Header 在低自由度任务上无效 |

### 7.3 逐模型交叉验证

| GPT-5.4 | 100% | 100% | 100% | 100% | 0 | 鲁棒 |
| GLM-Turbo | 100% | 100% | 100% | 100% | 0 | 鲁棒 |
| GLM-5.0 | 100% | 100% | 100% | 100% | 0 | 鲁棒* |
| Gemini-Flash | 100% | 100% | 100% | 100% | 0 | 鲁棒 |
| Gemini-Pro | 100% | 100% | 100% | 100% | 0 | 鲁棒 |

| Kimi | 100% | 100% | 100% | 100% | 0 | 鲁棒* |

| 类型 | 纯净代表 | 混合成员 | 判定依据 |
|------|---------|---------|---------|
| **指令鲁棒型** | Claude-Opus-4.6, GPT-5.4 | GLM-5.0-Turbo, Gemini-3.0-Flash, Gemini-3.0-Pro, Gemini-3.1-flash-lite | 全域独立 + 全格式等效 + 约束全遵循 |
| **格式敏感型** | DeepSeek-V3.2 | Claude-Haiku-4.5（仅 FE 域） | Pass@1 / 生成成功率随 prompt 格式显著波动 |
| **内容收敛型** | MiniMax-M2.7（全域） | Kimi-K2.5（BE/PY）, GLM-5.0（BE/PY）, Haiku（BE/PY） | 低自由度任务直接克隆 |

### 7.5 与传统能力层级的对照

| 传统层级 | 包含的行为类型 | 说明 |
|---------|-------------|------|
| Pro | 全部指令鲁棒 | 层级与行为一致 |
| Mid | 鲁棒 + 收敛混合 | **层级无法区分**：同为 Mid 的 Gemini-Flash（鲁棒）和 MiniMax（收敛）行为完全不同 |
| Lite | 格式敏感 + 鲁棒混合 | **层级无法区分**：同为 Lite 的 DeepSeek（格式敏感）和 flash-lite（鲁棒）行为相反 |

这说明 **Mid 和 Lite 层内部的行为异质性远大于层级间差异**。行为模式分类比能力层级对 prompt 工程实践的指导性更强。

### 7.6 三类模型的 Prompt 工程建议

| 类型 | Compact Header 可靠性 | 推荐策略 |
|------|---------------------|---------|
| **指令鲁棒型** | ✅ 高。Header 可直接用作硬约束 | 标准 Compact Header 即可，无需额外处理 |
| **格式敏感型** | ⚠️ 取决于格式。JSON 最差，Compact/Classical 更稳 | 避免 JSON spec 格式；优先使用 Compact 或 Classical；添加重试机制 |
| **内容收敛型** | ❌ 在低自由度任务上无效 | Header 差异不影响输出；仅在高自由度任务（如 FE 域）上有效 |

### 7.7 论文中的使用

这个分类框架适合放在 §6 Discussion 中，替代或补充原有的 Tier S/A/F 分层：

> *"基于 Pilot、EXP-A 和 EXP-B 的综合数据，我们观察到三类模型行为模式：指令鲁棒型（Instruction-Robust）、格式敏感型（Format-Sensitive）和内容收敛型（Content-Convergent）。这一分类与传统能力层级（Pro/Mid/Lite）正交——Mid 层同时包含全域鲁棒的 Gemini-3.0-Flash 和全域收敛的 MiniMax-M2.7，Lite 层同时包含格式敏感的 DeepSeek-V3.2 和条件鲁棒的 Gemini-3.1-flash-lite。行为模式分类揭示的是模型对结构化约束的响应机制差异，对 prompt 工程实践的指导比能力层级更直接。"*
