## 问题 2：结论 F3（变体敏感度分层）是否需要 follow-up 实验？

### 直接回答：需要，但不是验证"猜想"，而是量化机制。

我在报告中提出的"可能的解释"（BE/PY 任务实现方式固定，模型倾向于输出标准答案）确实是推测。要验证它，需要以下设计：

**Follow-up Experiment A：控制变量法确认"任务复杂度"的因果性**

| 设计 | 说明 |
|------|------|
| **假设** | BE/PY 任务的克隆是因为任务过于简单，而非模型不区分 prompt 变体 |
| **方法** | 对同一批克隆模型（GLM-5.0、Kimi-K2.5、MiniMax），给它们 BE 中等/进阶任务（如 JWT 认证、WebSocket 聊天），观察克隆是否消失 |
| **预期** | 如果克隆在进阶任务中消失 → 确认"任务简单"是原因；如果仍克隆 → 说明模型本身不敏感 |
| **成本** | 选 3 个克隆模型 × 3 个进阶任务 × 4 变体 = 36 次生成 |

**Follow-up Experiment B：prompt 干扰强度测试**

| 设计 | 说明 |
|------|------|
| **假设** | 克隆模型实际上"忽略了"约束 Header 部分，只看需求描述 |
| **方法** | 给克隆模型一个故意矛盾的 prompt：Header 说 FastAPI，需求描述说 Flask。如果输出遵循需求描述而非 Header，则证明模型忽略了 Header |
| **预期** | 确认克隆的根本原因——是"Header 被忽略"还是"任务太简单所以 Header 不影响输出" |
| **成本** | 3 个模型 × 2 个矛盾任务 × 4 变体 = 24 次生成 |

## 问题 3：Tier × Variant × Domain 全 100% 是否需要新实验？

### 直接回答：这不是猜想，这是天花板效应。全 100% 说明当前 12 个 pilot 任务的难度不够。

**不需要验证猜想，但需要调整实验设计。**

三因素交叉分析全 100% 的原因是明确的：
1. **Pro/Mid 层**：所有任务对这些模型来说太简单，约束遵循零挑战
2. **Lite 层**：DeepSeek 的失败是"文件缺失"（生成管线中断），不是"约束理解错误"；Gemini-3.1-flash-lite 也全通过

**解决方案不是 follow-up 实验，而是全量实验的设计调整**：

| 调整项 | v2 pilot 现状 | 建议调整 |
|--------|-------------|---------|
| 任务难度分布 | 基础 8 + 中等 4 = 12 个 | 增加进阶任务（v2 设计中每类有 4 个进阶，如拖拽排序、虚拟滚动、并发下载器） |
| 约束复杂度 | 大多数任务的约束一致（TW + React + 无外部库） | 增加"反直觉约束"变体（如 FE-04 的 plain_css，已证明是唯一有区分度的任务） |
| 约束冲突 | 无 | 增加边缘约束（如"只允许 CSS modules"或"禁止 useState"），提高 CCR 的区分度 |

**预期**：在 60 个任务中，进阶任务（12 个）应能将 Pass@1 拉低到 70-85% 区间，使统计检验有功效。

## 问题 4：克隆问题的 Follow-up 实验设计

### 直接回答：同意，克隆问题需要正面处理。

**克隆的核心事实**：

| 模型 | FE 克隆 | BE/PY 克隆 | 解读 |
|------|---------|-----------|------|
| MiniMax-M2.7 | 全克隆 | 全克隆 | 模型可能根本不区分 prompt 前缀 |

| GLM-5.0-Turbo | 独立 | 轻微（BE-01） | 接近正常 |

**建议的 Follow-up Experiment C：克隆排除实验**

| 设计 | 说明 |
|------|------|
| **目标** | 确认克隆是"平台缓存"还是"模型行为" |
| **方法 1** | 对 MiniMax-M2.7 重跑同一任务，但在 prompt 末尾加随机 seed 字符串（如 `[seed:a7b3c]`），看输出是否变化。如果加 seed 后输出变了 → 平台缓存问题；如果仍然相同 → 模型行为 |
| **方法 2** | 对 GLM-5.0，只重跑一个克隆的 BE 任务和一个独立的 FE 任务（共 8 次生成），在不同时间段执行，排除平台侧的缓存/去重 |
| **成本** | 方法 1：1 模型 × 2 任务 × 4 变体 = 8 次；方法 2：1 模型 × 2 任务 × 4 变体 = 8 次 |

**在论文中的处理（如果不做 follow-up）**：

如果时间不允许做克隆排除实验，论文中应该：
1. 在 Results 中如实报告克隆现象
2. 在分析组间差异（prompt variant effect）时，**排除克隆模型在克隆域的数据**
3. 在 Discussion 中讨论两种可能解释（平台缓存 vs 模型行为）并声明为 limitation

## 问题 5：Token 节省分析

### 准确度一致的前提下，token 节省情况如何？

这是 RQ1 最重要的数据。以下是从 240 份 prompt 文件直接测量的结果。

### 5.1 四组 Prompt 字符数（12 任务平均值）

| 变体 | 描述 | 平均字符数 | 相对 A 组 |
|------|------|-----------|----------|
| **A (NL)** | 自然语言基线 | **324** | — |
| **B (JSON)** | JSON Spec | **391** | **+20.7%**（更长） |
| **C (Compact)** | 英文紧凑 Header | **179** | **−44.6%**（节省） |
| **D (Classical)** | 古文 Header | **109** | **−66.3%**（节省） |

### 5.2 准确度 vs 成本矩阵

| 变体 | Pass@1 | CCR | 字符成本 | 结论 |
|------|--------|-----|---------|------|
| A (NL) | 97.6% | 0.990 | 324 字符 | 基线 |
| B (JSON) | 96.0% | 0.960 | 391 字符（**+21%**） | ❌ 更贵且更差 |
| C (Compact) | 99.2% | 0.992 | 179 字符（**−45%**） | ✅ **更便宜且等效/略优** |
| D (Classical) | 100% | 1.000 | 109 字符（**−66%**） | ⚠️ 字符最省，但需考虑 BPE |

### 5.3 逐任务 A vs C 的字符节省

| 任务 | A 字符 | C 字符 | 节省 | Pass@1 差异 |
|------|--------|--------|------|-----------|
| FE-01 | 306 | 139 | **−54.6%** | C ≥ A |
| FE-02 | 289 | 122 | **−57.8%** | C = A |
| FE-03 | 299 | 132 | **−55.9%** | C = A |
| FE-04 | 279 | 118 | **−57.7%** | C ≥ A |
| BE-01 | 291 | 156 | **−46.4%** | C = A |
| BE-02 | 343 | 214 | **−37.6%** | C = A |
| BE-03 | 310 | 179 | **−42.3%** | C = A |
| BE-04 | 320 | 189 | **−40.9%** | C = A |
| PY-01 | 322 | 186 | **−42.2%** | C = A |
| PY-02 | 384 | 248 | **−35.4%** | C > A |
| PY-03 | 394 | 258 | **−34.5%** | C = A |
| PY-04 | 346 | 210 | **−39.3%** | C = A |

**关键结论**：

> **Compact Header (C 组) 在 12 个任务上实现了平均 44.6% 的 prompt 字符节省，且没有任何一个任务出现 C 的 Pass@1 低于 A 的情况。**

这是 RQ1 最干净的回答：**节省了近一半的约束描述成本，准确度不降反升**。

### 5.4 重要说明：字符数 vs Token 数

上述数据使用的是**字符数**（char count），而不是 token 数。这是因为：
1. 不同模型使用不同的 tokenizer（GPT 系的 tiktoken vs Gemini 的 SentencePiece 等）
2. 在 WorkBuddy 平台上无法获取实际 token 消耗

但字符数和 token 数在英文内容上高度相关（约 1 token ≈ 4 chars）。**对于 D 组（古文），字符数的节省会被 BPE 对中文的低效编码部分抵消**——这是 v1 已经发现的结论（古文仅节省 4.6% token 而非 66% 字符）。

对于 C 组（英文 Compact），字符到 token 的转换比相对稳定，预期实际 token 节省在 **35-45%** 区间。

### 5.5 B 组（JSON）为什么最差

| 方面 | B vs A |
|------|--------|
| 字符成本 | **+20.7%**（JSON 的引号、括号、冒号是冗余字符） |
| Pass@1 | **−1.6%**（96.0% vs 97.6%，因 DeepSeek 解析失败） |
| 功能分 | **−0.12**（2.65 vs 2.77） |

**JSON 是最差的选择**：比自然语言更贵，比 Compact Header 更长，且在弱模型上稳定性更差。这直接回答了 reviewer 最可能问的"为什么不用 JSON"——因为 JSON 在每个维度上都不如 Compact Header。
