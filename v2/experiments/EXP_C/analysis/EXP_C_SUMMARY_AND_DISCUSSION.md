## 一、实验概况

### 设计
- **12 个 Benchmark 任务** (4 前端 + 4 后端 + 4 Python 工具)
- **3 种约束编码**: H (Header/DSL)、NLc (NL-compact)、NLf (NL-full)
- **3-stage 线性管线**: S1 Architect → S2 Implementer → S3 Auditor
- **7 种 agent 组合** (3 模型 × 3 stage 的单变量替换): RRR, CRR, SRR, RCR, RSR, RRC, RRS
  - R = Claude-Opus-4.6, C = Kimi-K2.5, S = DeepSeek-V3.2
- **总管线**: 12 × 3 × 7 = 252 条 (实际完成 246 + 6 个合理缺失)
- **关键设计特征**: 每个 stage 的 prompt 都携带完整的原始约束文本

### 产出数据
| 数据集 | 文件 | 行数 |
|--------|------|------|
| L2: S2 代码二值判定 | `constraint_binary_s2.csv` | 252 |
| L1: S1 四维度标注 (M/R/F/A) | `s1_mention_repr.csv` + `s1_fidelity_action.csv` | 252 |
| L3: S3 报告质量 | `s3_report_quality.csv` | 252 |
| Master 整合表 | `master.csv` | 252 (246 COMPLETE + 6 INCOMPLETE) |

## 二、确凿结论 (有充分数据支撑)

### 结论 1: Token 经济性 — Header 约束编码节省 ~25-30%

| 比较维度 | H vs NLf 节省 |
|---------|-------------|
| 约束段落字符数 | 71.4% |
| S1 完整 prompt | 25.5% |
| S2 完整 prompt | 47.7% |
| S3 完整 prompt | 26.1% |
| 3-stage 管线累计 | **30.5%** |

**交叉验证**: v1 实验 (古文 DSL) ~4.6%, v2 实验 (英文极简 DSL) ~25.6%, EXP-C (S1 单 stage) ~25.5%。三轮独立实验中英文 DSL 的节省比例高度一致 (~25-26%)。

**注意**: 约束在总 prompt 中的占比约 8%，所以 25-30% 的约束节省换算为总 prompt 节省约 2-6%。在多 agent 系统中，约束 token 与 agent 数线性增长，节省的绝对值等比放大，但百分比不变。

### 结论 2: 静态注入下 CSR 三组无差异

| 编码 | CSR-obj-S2 | 统计检验 |
|------|-----------|---------|
| H | 0.951 | KW H=0.60 |
| NLc | 0.944 | df=2 |
| NLf | 0.956 | **p>0.05 (ns)** |

**含义**: 当约束以原始文本直接注入每个 stage 的 prompt 时，三种编码方式的代码约束遵循率没有统计显著差异。

**重要限定**: 本实验中每个 stage 都携带了完整的原始约束文本，agent 不需要从上游 output 中推断约束。这是一个关键设计特征——见第四节讨论。

### 结论 3: 反直觉约束系统性失败 (与编码方式和架构无关)

| 约束类型 | 平均 CSR | 差异 |
|---------|---------|------|
| 常规 (C1/C4/C5/C6) | 0.991 | |
| 反直觉 (C2/C3) | 0.870 | Δ=0.121 |
| 统计检验 | | **p<0.001 ★★★** |

**具体表现**: 
- MC-FE-01 C2 (CSS Modules): 17/21 FAIL (81%)
- MC-FE-02 C3 (CSS Modules): 19/21 FAIL (90%)
- MC-FE-04 C3 (plain CSS): 11/21 FAIL (52%)

**域效应显著**: FE=0.907 < BE=0.970 < PY=0.977 (KW p<0.05)

**解读**: 这不是编码方式或管线架构的问题，而是模型对特定约束类型的理解能力局限。模型倾向于使用 inline style 对象而非 CSS Modules import——这是一种"默认行为偏好"对抗"显式约束"的现象。

## 三、需限定范围的发现

### 发现 A: S3 Output 大小差异

| 编码 | S3 平均大小 | NLf/H |
|------|-----------|-------|
| H | 2,444 chars | — |
| NLc | 3,754 chars | 1.54× |
| NLf | 5,019 chars | 2.05× |

**H vs NLf 统计显著 (p<0.001), 效应量 medium (d=0.513)**

**但经深入分析**:
- 排除修复代码块后，差异缩小到 ~11% (1,364 vs 1,513, p=0.028)
- 2× 差异的主驱动因素是: NLf 条件下 FAIL 更多 → S3 需要输出修复代码
- **不是所有任务都 > 1.0**: PY-01=0.42×, PY-03=0.86×, PY-04=0.74×
- 前端/后端放大明显 (FE=2.73×, BE=2.55×), Python 反转 (PY=0.68×)
- **结论**: 差异由实验设计 (S3 prompt 要求输出修复代码) 驱动，非编码方式固有属性

### 发现 B: Stage Attribution

所有 combo vs RRR 基线的 CSR 差异**均不显著 (ns)**。单个 stage 的模型替换不产生可检测的 CSR 影响。

### 发现 C: S3 Auditor 判定偏差

S3 auditor 的约束判定与 L2 客观规则的一致性:
- 平均 5.72/6 (95.3%)
- Opus S3 偏宽松 (29 lenient vs 11 strict)
- DeepSeek S3 轻微偏严 (9 lenient vs 11 strict)

## 四、实验设计的已知局限

### 局限 1: 线性链不是主流多 agent 架构

我们的 A→B→C 线性链在业界和学界均非主流。当前主流架构:

| 架构 | 代表 | 信息共享方式 |
|------|------|------------|
| Blackboard (共享画布) | LbMAS (Han & Zhang, 2025) | 所有 agent 读写同一画布 |
| 并行隔离 | Cursor 2.0 | 多 agent 独立工作，最后 diff 合并 |
| Orchestrator-Worker | CrewAI / LangGraph | DAG/状态图编排，共享 state |
| 单 Agent 闭环 | Copilot / Windsurf | 无多 agent 通信 |

纯线性串行在实践中缺乏代表性——如果任务可以线性分解为 A→B→C，通常不如让一个更强的 agent 做全链路。

### 局限 2: 每个 stage 注入完整原始约束

这是最核心的设计问题。本实验中，S2 的 prompt 同时包含:
- S1 的 output (技术方案)
- **原始约束文本的完整副本**

这意味着 S2 不需要从 S1 的 output 中"理解"约束——它直接从 prompt 看到了原文。因此:
- "Naturalization 不影响下游 CSR" 这个结论是 **trivial** 的
- "编码方式对 CSR 无差异" 也可能是因为约束没有机会在传递中衰减

如果设计改为 "约束只出现在 S1 prompt 中，S2 只能从 S1 的 output 中获取约束信息"，编码方式的影响可能会显现——因为 naturalized 后的约束在信息保真度上可能不如结构化原文。

### 局限 3: 未涉及约束传递衰减

由于局限 2，本实验**实际上没有测试约束在 agent 间传递时的衰减**。标题中的"多 agent"主要指管线结构，而非约束传播。

## 五、研讨纪要 — 关于实验方向的辩论

以下记录了实验分析过程中的关键讨论，保留原始论点。

### 5.1 Token 节省的量级问题

**初始观点**: "多 agent 消耗 4-220× 单 agent token，25-30% 的约束节省在这个量级下微不足道"

**纠正**: 这是错误的推论。多 agent 的 token 放大是叠加态，约束节省也等比放大。但约束在总 prompt 中的占比 ~8%，所以 25-30% 约束节省 ≈ 2-6% 总 token 节省，这个比例不随 agent 数变化。

**结论**: 不是"微不足道"，但也不是量级性的改变。实际影响取决于约束文本长度和 agent 数量。

### 5.2 静态约束 vs 动态约束

**静态约束** (预定义的项目规范、技术栈限制):
- 可以一次性编写为 DSL，注入每个 agent
- EXP-C 已证明: 静态注入下 CSR 不衰减，token 节省 ~25-30%
- 在所有架构下都适用 (约束遵循只取决于单 agent 能力，与拓扑无关)

**动态约束** (运行时产生的派生决策):
- 例: Agent A 决定"用 gRPC"，这个决策对 B 构成约束
- 好的 orchestrator 可以自动提取 A 的决策并结构化广播给 B/C → 退化为静态注入
- 这是**工程设计问题**，不是需要大量实验的研究问题

### 5.3 "DSL 全链路"方案的成本分析

**设想**: 多 agent 中间环节全部强制 DSL 输入输出

**分析**:
- 对约束/规则部分: DSL 可以节省 ~70% 的约束 token → free lunch (静态预编写)
- 对方案/代码/讨论部分: 无法用 DSL 表达 → 无法压缩
- NL→DSL 的动态转换: 需要额外 LLM 调用，成本可能反超节省
- **结论**: 静态约束用 DSL 是 free lunch; 动态约束来回转换不划算

### 5.4 四种主流架构下是否需要做实验

**辩论结果**: 对于静态约束不需要做四架构对比实验。但建议做一个小规模 "context dilution test"——验证约束在长 context (50K+ token) 的 blackboard 中是否会被"淹没"导致 CSR 下降。如果不下降 (大概率)，则静态约束在所有架构下都不是问题。

### 5.5 实验的核心价值重新评估

**EXP-C 的确凿贡献**:
1. 验证了 agent-to-agent 场景的 token 节省比例 (~25-30%) 与 human-to-agent (v2) 一致
2. 首次系统记录了 naturalization 现象
3. 发现了反直觉约束的系统性失败模式
4. 建立了可复用的方法论框架 (CSR 评分体系、衰减曲线)

**EXP-C 未能回答的问题**:
1. 在约束仅通过上游 output 传递 (非直接注入) 的场景下，编码方式是否影响 CSR?
2. 在 blackboard 等非线性架构中，约束传播的衰减规律是什么?
3. 不同模型的 naturalization 行为差异是否导致下游信息损耗?

## 六、工程最佳实践建议 (基于实验数据)

1. **将项目级约束用结构化格式维护**: 编写一份 compact header 或 DSL 约束列表，每个 agent 启动时注入。一次性编写，每次节省 ~25-30% 约束 token。

2. **不要依赖 agent 间的自然语言传递约束**: 我们观察到，当 S1 Architect 接收 Header 编码约束时，Opus 和 Kimi 会 100% 将其 naturalize 为自然语言写入方案中，DeepSeek 则保留结构化形式 (Naturalization Rate: Opus=100%, Kimi=100%, DeepSeek=0%)。这意味着如果下游 agent 只能从上游 output 中获取约束，结构化编码的优势在第一个环节就会消失。因此约束应作为不变量直接注入每个 agent，而非期望通过上游传递。（注: 本实验中由于采用了静态注入设计，naturalization 未实际影响下游 CSR——但它解释了为什么静态注入是必要的。）

3. **对于运行时决策，orchestrator 应负责提取和广播**: 当上游 agent 做出技术决策后，orchestrator 应将其结构化并注入下游 agent 的 prompt，使其退化为"静态注入"。

## 七、后续方向建议

| 方向 | 价值 | 成本 | 优先级 |
|------|------|------|--------|
| Context dilution test | 验证长 context 下约束是否被淹没 | 低 (~50 次调用) | P1 |
| 约束仅通过上游传递的实验 | 测试 naturalization 是否真正导致 CSR 衰减 | 中 (~150 次调用) | P1 |
| 四架构对比实验 | 学术完整性，但可能结论同质 | 高 (~500 次调用) | P2 |
| 动态约束传播 | 学术新颖性高，但实用场景可能窄 | 中高 | P3 |
