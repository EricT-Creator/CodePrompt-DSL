## 关键发现

### 1. DSL 压缩在英文约束下成立
- 对大多数可信模型，A组（英文自然语言）与 D组（英文极简DSL）几乎无显著差异
- 因此“更短的工程约束表达”是成立的，且能带来约 25% token 节省

### 2. 极简古文不是高性价比方案
- F组在多数模型上更不稳定
- 中文 BPE 下，古文表达并没有带来足够明显的 token 优势
- 对关键业务生成，不建议优先使用 F 组

### 3. Hunyuan-2.0-Instruct 应谨慎解读，而不是直接判为流程污染
- 该模型已经重跑过一遍，当前结果不再按“未测试”或“纯模板污染”处理
- 更合理的解释是：它在基础指令理解、需求吸收和稳定执行上存在明显短板
- 因此它的结果可以保留，但在对外结论中应加注“模型理解能力偏弱”说明

### 4. 不可信模型不纳入正式排行
- `claude-haiku-4.5`
- `hunyuan-2.0-thinking`

这两组当前只能保留为历史产物，不能继续用于正式排行、模型推荐或方法有效性证明。

## 对选题的回扣：DSL / 古文分别适合落在哪类模型上？

### 1. DSL 最适合落在“已经会稳定写代码”的模型上

从当前可信结果看，D 组真正体现价值的前提不是“模型被 DSL 教会了写代码”，而是：

> **模型本来就具备较强的结构化代码生成能力，DSL 只是把高频显式约束压缩得更短、更稳、更容易复用。**

因此，最适合作为 CodePrompt-DSL 主承载对象的，是下面两类模型：

**第一类：顶级、结构约束吸收最稳定的模型**
- `gemini-3.0-pro`
- `gpt-5.4`

**第二类：中文/本地化场景下更现实的承接模型**
- `glm-5.0-turbo`
- `kimi-k2.5`
- `minimax-m2.7`

这些模型的共同点是：
- A→D 几乎无损
- 能把 DSL 当作正式约束层使用
- 更适合做真实生产中的 Prompt 模板化与成本优化

### 2. 弱模型不是 DSL 的主要舞台

`deepseek-v3.2` 与 `gemini-3.1-flash-lite` 在 A/D/F 三组都低分，说明编码形式不是它们的主要瓶颈。

这意味着：
- **CodePrompt-DSL 不是弱模型增强器**
- 如果模型基础结构化代码能力不足，换成 DSL 也不会把它抬成可用生产级

### 3. 古文更适合作为研究支线，而不是产品主线

当前可信结果支持的最稳妥说法是：
- 古文编码并非在所有模型上都会灾难性失效
- 但它的稳定性、可解释性、以及性价比都不如 D 组清晰
- 因而它更适合作为“语义压缩边界测试”而非“默认生产方案”

换句话说：

> **如果项目要落地，主线应是 DSL Header；如果项目要保留研究趣味与反直觉价值，古文可以作为对照组继续存在。**

### 4. 选题最值得保留的核心价值

这轮实验真正为选题留下来的，不是“古文 Prompt 真的更强”，而是三点：

1. **显式工程约束可以被压缩成紧凑 Header，并在强模型上基本无损**
2. **模型能力越强，越能把编码形式差异消化掉**
3. **Prompt 优化不该把所有内容一锅压缩，而应把“闭集约束”与“开放需求”分开处理**

## 正式结论

1. **正式比较应仅基于可信模型结果**，并将 `hunyuan-2.0-instruct` 作为“谨慎保留结果”单独说明
2. **CodePrompt-DSL 的主目标模型**：优先考虑 `gpt-5.4`、`gemini-3.0-pro`，其次是 `glm-5.0-turbo`、`kimi-k2.5`、`minimax-m2.7`
3. **编码方式建议**：生产默认优先使用 D 组；A 组可作为最稳的自然语言基准；F 组保留为研究性对照，不建议充当默认生产编码
4. **中文场景建议**：若关心本地化与中文生态，优先考虑 `glm-5.0-turbo` 与 `kimi-k2.5`，`minimax-m2.7` 可作为稳定补充
5. **弱模型使用建议**：`deepseek-v3.2`、`gemini-3.1-flash-lite` 不应作为 CodePrompt-DSL 方法有效性的主证明对象
6. **对外发布要求**：必须明确标出“不可信模型”与“谨慎保留模型”，并注明当前任务级机制解释仍受评估器字面匹配偏置影响
