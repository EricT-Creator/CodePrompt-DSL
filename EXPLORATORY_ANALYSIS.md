## 二、6 个稳健现象，以及 3 个需要收紧的解释

> **重要说明**：本文件的统计结论来自现存的 accuracy JSON；当前仓库缺少 9 个可信模型的 `.tsx` 原始源码，因此凡是涉及“它到底有没有实现，只是换了写法吗”的判断，都只能写成**高概率推断**，不能当作已完成的人工复核结论。

### 现象 1：「编码结果高度稳定」—— 5 个模型在 A/D/F 三组完全同分

| 模型 | A=D=F? | 说明 |
|------|--------|------|
| Gemini-3.0-Pro | ✅ 全部 30 个任务 A=D=F | 每一个任务、每一个维度、每一种编码方式得分完全相同，且全满分 |
| GPT-5.4 | ✅ 全部 30 个任务 A=D=F | 唯一失分点是 T06 的 techStack，但三种编码方式失分位置完全一致 |
| Gemini-3.0-Flash | ✅ 全部 30 个任务 A=D=F | 失分集中在 style 维度，且三组一致 |
| DeepSeek-V3.2 | ✅ 全部 30 个任务 A=D=F | 虽然得分低，但三组失分模式完全一致 |
| Gemini-3.1-Flash-Lite | ✅ 全部 30 个任务 A=D=F | 同上 |

**稳妥表述**：对这 5 个模型来说，在当前评估体系下，Prompt 编码形式从 A→D→F 没有造成可见分数波动。

**不宜直接上升为**：它们已经“完全不受表面形式影响”。

原因很简单：现有数据能证明的是“**分数不变**”，但还不能证明“**内部理解机制完全等价**”。

### 现象 2：「古文衰减主要集中在 Claude Opus」—— 但原因解释要克制

| 模型 | A→D 变化 | A→F 变化 | 当前口径 |
|------|---------|---------|-----------|
| Claude Opus 4.6 | -0.10 | **-0.40** | 显著下滑 |
| GLM-5.0-Turbo | 0.00 | -0.10 | 轻微下滑 |
| Kimi-K2.5 | 0.00 | -0.10 | 轻微下滑 |
| MiniMax-M2.7 | 0.00 | 0.00 | 无明显变化 |
| GPT-5.4 | 0.00 | 0.00 | 无明显变化 |
| Gemini-3.0-Flash | 0.00 | 0.00 | 无明显变化 |
| Gemini-3.0-Pro | 0.00 | 0.00 | 无明显变化 |

可以确认的事实是：**Claude Opus 是强模型里 F 组下滑最明显的一个。**

但下面这类话，现在还不够严谨：
- “这是因为它不理解古文语义”
- “中国模型天然更懂古文，所以不会掉分”

**当前更稳妥的写法**：Claude Opus 在 F 组的功能维度失分更集中；至于这究竟是古文理解问题、关键词命中问题，还是两者叠加，仍需源码级复核。

### 现象 3：「T07 / T09 形成稳定分裂」—— 这是结果事实，不等于已证实的能力鸿沟

T07 ImageGallery 的功能关键词是 `grid, lightbox, lazy`；T09 DataTable 的关键词是 `sort, paginat, search`。

| GLM | 0 (67%) | 0 (67%) | 0 (67%) | ❌ |

| GPT-5.4 | **1 (100%)** | **1 (100%)** | **1 (100%)** | ✅ |

#### T09 结果分布

| 模型 | T09 features 通过? | 实际命中 | 缺失 |
|------|-------------------|---------|------|
| Opus | ❌ | sort, search | **paginat** |
| GLM | ❌ | sort, search | **paginat** |

**可以确认的结果事实**：
- GPT-5.4 与 Gemini 系列在 T07 / T09 上稳定命中目标关键词
- Opus / GLM / Kimi / MiniMax 在这两个任务上稳定漏掉关键字面词

**现在还不能确认的事**：
- 这些低分模型是不是其实做了功能，但没写到 `lightbox` / `paginat`
- 因而，这里不能直接下结论说它们“不会做灯箱 / 不会做分页”，也不能反过来说它们“一定做了只是换词”

### 现象 4：「评估器确实存在字面关键词偏置」—— 这是目前最硬的一条结论

结合评估脚本，可以直接确认当前评分机制高度依赖**字面字符串匹配**：

- techStack 会直接检查 `import React`
- techStack 会直接检查 `: React.FC`
- T07 只看 `grid`, `lightbox`, `lazy`
- T09 只看 `sort`, `paginat`, `search`
- 功能命中率需达到 75% 才给 1 分

这意味着：
- 如果代码使用了新的 JSX runtime，不写 `import React`，可能被误判
- 如果代码是 TypeScript，但没有写成 `: React.FC` 风格，也可能被记成 `TS:false`
- 如果图库预览实现写成 `modal` / `overlay` / `preview`，却没写 `lightbox`，也会丢分
- 如果分页实现写成 `page` / `pageSize` / `currentPage`，却没写 `paginat` 前缀，仍会丢分

> 🔑 这条不是猜想，而是**已确认的评估机制事实**。

### 现象 5：「模型均分呈双峰」—— 但不能写成“能力全有或全无”

模型得分分布存在明显的**双峰结构**：

```
5.0  ████████ Gemini-Pro (5.00)
4.9  ██████ GPT-5.4 (4.90)
4.8  ████ GLM/Kimi (4.77)
4.7  ███ MiniMax (4.70)
4.6  ██ Opus (4.63)
4.3  ██ Flash (4.30)
     ... 中间没有模型落在 2.0-4.2 ...
1.3  █ DeepSeek (1.30)
1.1  █ Lite (1.10)
```

**可以确认的事实**：没有任何模型的全局均分落在 1.4 ~ 4.2 之间。

**需要收紧的解释**：这并不自动等于“代码生成能力是全有或全无”。

原因在于：
- DeepSeek-V3.2 的 T01 仍能拿到 `4/5`
- Gemini-3.1-Flash-Lite 在部分任务上也不是纯零分，而是“React 风格输出 + 结构约束不合格”

更准确的表述应该是：

> **当前实验中的“模型平均分”呈双峰分布，但任务级表现并非纯两态开关。更像是模型在“结构化工程约束适配能力”上存在明显分层。**

### 现象 6：「GPT-5.4 的 T06 异常」—— 结果明确，成因暂作工作假设

GPT-5.4 在 T06（MarkdownEditor）上，所有三组都丢了 techStack 分（`TS:false`），但这是它全局唯一的失分点。

| 维度 | GPT-5.4 T06 A | GPT-5.4 T06 D | GPT-5.4 T06 F |
|------|--------------|--------------|--------------|
| techStack | **0** (TS:false) | **0** (TS:false) | **0** (TS:false) |
| form | 1 | 1 | 1 |
| style | 1 | 1 | 1 |
| dep | 1 | 1 | 1 |
| features | 1 | 1 | 1 |

**可以确认的事实**：
- GPT-5.4 的 T06 在三组都被 techStack 维度绊倒
- 除此之外，它没有别的失分点

**当前只能写成工作假设**：
- 它可能用了未被脚本识别的 TS 写法
- 或者采用了不包含 `import React` / `: React.FC` 的现代 React 书写方式

由于源码缺失，这里不能把成因写死。

## 三、3 个猜想的修订版判定

### H1：「词汇选择偏好假说」

> **修订表述**：模型之间在 T07 / T09 上的分差，至少部分可能来自“词汇选择偏好”与评估关键词之间的错位，而不一定完全等于功能能力差距。

**当前证据强度**：高

**已经确认的部分**：
- 评估器确实存在字面关键词偏置
- 多个强模型在 T07 / T09 上稳定出现相同漏词模式

**尚未确认的部分**：
- 这些模型是否真的完成了等价功能实现

**当前结论口径**：
- 可以写成：**“高概率成立，待源码复核确认”**
- 不宜写成：**“已经证实这些模型只是换词，没有能力差异”**

### H2：「能力阈值假说」

> **修订表述**：当前实验显示模型均分存在明显双峰，但更像是“结构化工程约束适配能力”分层，而不是绝对的“能力全有或全无”。

**当前证据强度**：中高

**已经确认的部分**：
- 均分双峰分布成立
- 弱模型在 techStack / form / style 上存在系统性失分

**尚需保留的部分**：
- 低分模型到底是“不会写”，还是“写法与检测规则错位”

**当前结论口径**：
- 可以写成：**“均分双峰成立，但任务级不是纯两态”**
- 不宜写成：**“代码生成能力本质上是全有或全无”**

### H3：「古文免疫假说」

> **修订表述**：在当前任务集与评分方式下，古文编码衰减并非普遍现象；但具体成因仍不能脱离源码复核直接定性。

**当前证据强度**：中高

**已经确认的部分**：
- 多个主流模型 A = D = F，或 F 仅轻微下降
- Claude Opus 的 F 组下滑最明显

**尚未确认的部分**：
- Claude Opus 的 F 组下滑，究竟主要来自古文理解问题、字面关键词偏置，还是两者叠加

**当前结论口径**：
- 可以写成：**“结果层面成立，机制层面待验证”**
- 不宜写成：**“Claude 的古文下滑已经确定由语义映射不足造成”**

## 四、附加发现：维度级分析

### 4.1 各维度的全模型通过率

| 维度 | 强模型群(7个) | 弱模型群(2个) | 说明 |
|------|-------------|-------------|------|
| techStack | 97.6% | 3.3% | 弱模型几乎全不通过 |
| form | 99.0% | 0% | 弱模型几乎不满足 `export default` 形式 |
| style | 96.2% | 0% | 弱模型几乎都没命中 Tailwind / 样式规则 |
| dep | 100% | 100% | **所有模型 100% 通过** |
| features | 80.5% | 6.7% | 强模型之间的主要分差集中在功能维度 |

> 🔑 **dep（依赖约束）维度完全没有区分度。** 所有 9 个模型在所有 270 次测试中 dep=1，说明“不引入第三方库”在当前任务集上更像默认行为，而不是有效的能力筛选项。

### 4.2 当前评估更接近“工程约束 + 功能关键词命中测试”

在 7 个强模型中，techStack / form / style / dep 的通过率都在 95%+。模型间真正拉开分差的，几乎都来自 features 维度。

这意味着当前评估体系更像：
1. **结构化工程约束检测**（React + TS + export + Tailwind）
2. **功能关键词命中检测**

它**还不是**一个完整的代码质量评估体系。比如下面这些目前都没有被测进去：
- 交互体验是否顺滑
- 状态管理是否合理
- 边界条件是否处理
- 组件可维护性与可读性

## 五、修订后的结论与建议

### 5.1 目前可以较稳地写进正文的结论

1. **英文 DSL（D 组）在多数可信模型上基本不损失准确度**，因此“压缩 token 而不明显降分”的方向是成立的
2. **古文编码（F 组）不是普遍灾难**，但不同模型的稳定性确有差异
3. **当前评分体系存在明确的字面关键词偏置**，尤其体现在 `lightbox`、`paginat`、`import React`、`: React.FC` 等检测点
4. **模型均分呈双峰**，但更适合解释为“结构化约束适配能力分层”，而不是“能力全有或全无”

### 5.2 暂时不要写成既定事实的结论

1. “Opus / GLM / Kimi / MiniMax 在 T07 / T09 上其实都做对了，只是换了词”
2. “DeepSeek / Lite 只是格式问题，不是能力问题”
3. “Claude Opus 的古文下滑已经确定是古文语义映射弱导致”

这些说法都还差最后一步：**源码级人工复核**。

### 5.3 对评估体系的直接改进建议

1. **扩充功能同义词表**
   - 例如 T07 不应只认 `lightbox`
   - T09 不应只认 `paginat`
   - 每个功能点至少应设计 3 组等价表达

2. **修正 techStack 检测规则**
   - 不应把 `import React` 与 `: React.FC` 当作近似必要条件
   - 应允许现代 React / TypeScript 的多种合法写法

3. **替换 dep 维度**
   - 当前 100% 通过，没有区分力
   - 可替换为更有效的结构性指标，例如状态管理、事件处理完整性、空态与边界情况覆盖等

4. **补做源码级人工复核**
   - 优先恢复并检查 T06 / T07 / T09
   - 这三类任务最能区分“真没做”与“做了但漏词”

### 5.4 对 CodePrompt-DSL 方法论的启示

1. **DSL 编码方案的核心价值仍然成立**：至少在当前可信结果中，A→D 基本无损
2. **真正拖累结论可信度的，主要不是 DSL 本身，而是评估口径过于字面化**
3. **下一阶段研究重点应该从“模型能不能看懂 DSL”转向“如何让评估更公平、更接近真实实现”**
