## 失分分析

### 失分任务清单（funcScore=0）

| 任务 | A组 | D组 | F组 | 原因分析 |
|------|-----|-----|-----|---------|
| **T01_TodoApp** | ❌ | ❌ | ❌ | `done`替代`complete`，未完全匹配关键词 |
| **T04_ShoppingCart** | ✓ | ✓ | ❌ | F组古文缺少价格计算的明确关键词 |
| **T06_MarkdownEditor** | ✓ | ✓ | ❌ | F组古文中未明确实现markdown预览 |
| **T07_ImageGallery** | ❌ | ❌ | ❌ | 缺少`lazy`关键词（懒加载未显式编码） |
| **T09_DataTable** | ❌ | ❌ | ❌ | 缺少`paginat`关键词（分页机制不清晰） |

### 失分根本原因

1. **功能关键词匹配严格性**
   - 评估规则要求关键词至少75%命中
   - Claude-Haiku-4.5 有时使用同义词（如`done`vs`complete`）
   
2. **古文编码理解差异**
   - F组相对A/D组下降0.20分
   - 中文古文的语义浓度导致部分需求表达模糊

3. **隐性功能实现**
   - 某些功能已正确实现但关键词不明确出现
   - 如懒加载可能通过逻辑实现但未用`lazy`变量命名

## 模型特征识别

### 代码风格特点

从生成的30个文件中可识别的特征：

```typescript
// 1. 变量名更短更简洁
add() vs addTodo()
del() vs deleteTodo()  
pwd vs password
qty vs quantity

// 2. 条件表达式更紧凑
filter === 'active' ? !t.done : t.done
// vs
if (filter === 'active') return !t.completed

// 3. 样式类定义更简洁
text-sm vs text-base
px-2 vs px-3
```

### 模型优势

- ✅ **轻量级**：生成速度快，内存占用低
- ✅ **DSL理解**：A和D组表现对称（均4.70），说明对紧凑编码理解能力强
- ✅ **古文表现**：F组4.50分，相对Hunyuan-Thinking的3.70有明显优势
- ✅ **代码可读性**：简洁清爽的风格适合快速原型开发

### 模型劣势

- ❌ **功能准确度**：某些任务的关键词实现不够完整
- ❌ **复杂组件**：在高交互组件上有失分（T07 Gallery、T09 DataTable）
- ❌ **古文编码**：虽然优于Thinking版，但仍有0.20分与D组的差距

## 编码方式对比（A vs D vs F）

### 核心发现

```
Encoding Robustness:
A组 = D组 = 4.70  (完全等价，差异=0)
F组 = 4.50        (差异=-0.20)

分析：
• Claude-Haiku-4.5对英文编码（自然语言和DSL）理解完全对称
• 这说明该模型对极简DSL的压缩形式零适应成本
• 古文编码下降0.20分是模型对中文古文理解的真实差异，非编码形式问题
```

### 建议

| 场景 | 推荐编码 | 理由 |
|------|---------|------|
| 一般用途 | A组（自然语言） | 最直观，避免歧义 |
| Token节省 | D组（DSL） | 节省25%+，效果等同 |
| 高精度中文 | 不推荐F组 | 古文理解率仅90% |
