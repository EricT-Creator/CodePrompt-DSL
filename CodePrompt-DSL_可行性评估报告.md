## 一、Executive Summary

**一句话定位：** 为高频使用 LLM 代码生成的开发者/平台，提供一种结构化 Prompt 中间层（DSL Header），用以压缩重复工程约束、降低 token 成本、提升约束遵循稳定性。

**总体评价：** 作为**开源研究实验**，这个项目有真实的技术探索价值。但需要正视一个核心矛盾，并据此调整项目定位和架构设计。

**PCV 评分：8/24（详见下文）** → 中等偏弱机会，作为开源实验值得推进，但距离创业产品还需要大幅强化价值主张。

## 二、你提出的核心矛盾：NL → DSL 的转换成本

### 问题陈述

你准确地识别了最关键的矛盾：

> "用户用自然语言输入 → 系统把自然语言转换成 DSL → 再把 DSL + 剩余 NL 发给模型"
>
> 这个转换本身如果需要调用 LLM，那转换消耗的 token 可能 ≥ DSL 节省的 token。

### 深度分析

这个担忧是**完全合理的**，而且是决定项目能否成立的分水岭。让我把不同场景拆开来看：

#### 场景 A：单次调用（One-shot）

```
传统路径：  用户 NL (200 tokens) → LLM → 代码
DSL 路径：  用户 NL (200 tokens) → [转换 LLM: 200 in + 50 out] → DSL+NL (150 tokens) → LLM → 代码

传统总消耗：200 tokens input
DSL 总消耗：200 + 50 + 150 = 400 tokens（反而翻倍！）
```

**结论：单次调用场景下，如果用 LLM 做转换，项目不成立。** ❌

#### 场景 B：重复模板 + 本地规则转换（无需 LLM）

```
用户设定一次约束：React + TS + 无外部库 + 移动端
系统通过本地规则/正则/UI表单 → 生成 DSL Header（零 token 成本）
之后每次调用：DSL Header (30 tokens) + NL需求 (100 tokens) 替代原来的 NL (200 tokens)

传统每次：200 tokens × N 次
DSL 每次：130 tokens × N 次
```

**结论：当约束通过非 LLM 方式提取（UI 选择、本地模板、规则引擎），且高频复用时，项目成立。** ✅

#### 场景 C：平台/系统级集成

```
平台预设约束 Profile（如 Cursor 的 .cursorrules）
DSL Header 由系统自动注入，用户无感
每次请求节省重复的约束描述
```

**结论：最有价值的场景——约束是系统级的、重复的、可复用的。** ✅✅

### 核心结论

| 转换方式 | 单次调用 | 高频复用 | 平台集成 |
|---------|---------|---------|---------|
| LLM 转换 | ❌ 不划算 | ⚠️ 摊薄后可能勉强 | ⚠️ 取决于频率 |
| 本地规则/模板 | ✅ 零成本 | ✅✅ 非常划算 | ✅✅✅ 最优 |
| UI 表单选择 | ✅ 零成本 | ✅✅ 非常划算 | ✅✅✅ 最优 |

**关键修正建议：项目应明确将 "NL → DSL" 的转换定位为本地/规则化转换，而非 LLM 转换。DSL 的价值在于"一次定义，多次复用"的约束模板化，而非实时翻译。**

## 三、Problem Assessment（问题评估矩阵）

| 维度 | 评级 | 影响 | 证据 |
|------|------|------|------|
| **频率** | Weekly-Daily | **Med** | 重度 AI 编程用户每天多次调用，但大部分个人用户是偶发使用 |
| **严重度** | Nice-to-have | **Low** | token 成本对个人开发者不构成痛点（GPT-4o 已大幅降价）；对平台/企业级可能是 Important |
| **意识度** | Aware but not seeking | **Low** | 开发者知道 prompt 冗长，但很少主动寻找 DSL 化方案 |
| **预算** | No dedicated budget | **Low** | 个人不会为 prompt 优化付费；企业可能会，但优先级低 |

**Problem Score：0 High / 1 Med / 3 Low = 偏弱**

### 为什么偏弱？

1. **Token 价格趋势向下。** 2024-2026 年间，主流模型价格下降了 10-50 倍。GPT-4o-mini 的 input 价格是 $0.15/M tokens。节省 15% 约等于节省了……几乎可以忽略的费用。
2. **真正的痛点不是 token 数量，而是约束遵循不稳定。** 这才是开发者真正 frustrated 的地方。
3. **市场上已有类似替代：** `.cursorrules`、`CLAUDE.md`、系统 prompt 模板——这些都在做类似的事情，且用户已经在用。

## 六、竞品分析

| 竞品/替代方案 | 类型 | 优势 | 劣势 | CodePrompt-DSL 的差异 |
|-------------|------|------|------|---------------------|
| `.cursorrules` | 直接竞品 | 已有生态、自然语言、零学习成本 | 冗长、无结构 | 更紧凑、结构化 |
| `CLAUDE.md` | 直接竞品 | 同上 | 同上 | 同上 |
| System Prompt 模板 | 间接竞品 | 灵活、自然语言 | 无标准化 | 标准化、可计量 |
| Prompt 缓存（API 特性） | 替代方案 | 原生支持、零改造 | 需要 API 层面 | 正交关系——可叠加 |
| Agent 框架自动管理 | 趋势替代 | 完全自动化 | 黑盒 | DSL 可作为 Agent 的约束注入层 |

### 关键发现

**Prompt Caching（提示缓存）是你最大的竞争威胁。**

OpenAI、Anthropic 都已支持 prompt caching：相同前缀的 prompt 在后续调用中自动缓存，input token 成本降低 50-90%。这意味着即使你重复发送完全相同的长自然语言约束，缓存机制也能帮你省掉大部分成本——**不需要 DSL**。

## 七、核心优势与需要提升的地方

### ✅ 项目做得好的地方

1. **科研方法论扎实。** paired t-test、benchmark 设计、统计验收标准——这在开源实验中非常少见，体现了严谨态度。
2. **边界清晰。** 明确了 "非目标"，没有过度承诺。
3. **分阶段推进。** Phase 1/2/3 的规划合理。
4. **核心洞察有价值。** "显式工程约束是高度重复的、可以结构化的" 这个观察是正确的。

### ⚠️ 需要重大调整的地方

#### 1. 重新定义价值主张（最重要）

**当前价值主张：** "省 token"

**问题：** token 价格下降 + prompt caching → 这个价值正在快速贬值

**建议调整为：**

> CodePrompt-DSL 的核心价值不是省 token，而是**约束的标准化、可复用、可审计**。

具体来说：
- **标准化：** 团队内统一的约束规范（类似 `.editorconfig` 对编辑器做的事）
- **可复用：** 一次定义，跨会话、跨工具复用
- **可审计：** 约束是显式的、可追踪的，而非散落在自然语言中
- **可组合：** 约束可以像乐高一样拼装（mobile + dark-mode + no-ext-lib）

这比 "省 15% token" 有意义得多。

#### 2. 明确 DSL 生成路径（你提出的核心矛盾）

**必须解决的架构问题：**

```
方案 A（推荐）：本地规则引擎 / UI 表单
  用户通过下拉/勾选 → 本地生成 DSL Header → 零 token 成本

方案 B（推荐）：预设模板 Profile
  用户选择/编辑一次 Profile → 保存为 .codeprompt 文件 → 自动注入

方案 C（不推荐作为默认）：LLM 转换
  仅用于"从已有自然语言 prompt 中一次性提取约束"这个初始化场景
  不应用于每次调用的实时转换
```

#### 3. Benchmark 设计需要增强

当前 20 个 React+TS 任务太单一。建议：

- 增加对比组 C：**自然语言 + Prompt Caching**（这是真正的竞品基线）
- 增加对比组 D：**自然语言 + `.cursorrules` 文件**
- 评估维度增加"约束遵循率"的量化指标（不只是 0/1）

#### 4. 古文灵感 → 需要彻底剥离

文档中多处提到"古文"作为灵感来源，但这会：
- 混淆项目定位（让人以为是中文特色项目）
- 增加不必要的文化解释成本
- 与实际的 DSL 设计无关

建议：项目叙事中可以提一句灵感来源，但名称、文档、README 中不应出现"古文"相关表述。项目名建议改为更直接的名字，如 `PromptSpec` 或 `CodeSpec`。

#### 5. 生态卡位策略

作为开源项目，最重要的不是技术实现，而是**成为标准**。建议：

- 定义 `.codeprompt` 或 `.codespec` 文件格式规范
- 做成 VS Code 插件（GUI 配置 + 自动注入）
- 提供 CLI 工具和 API SDK
- 与 Cursor / Windsurf / Continue 等工具集成

| 维度 | 当前状态 | 创业所需 | Gap |

| **付费意愿** | 无 | 明确 | ❌ 需要验证 PMF |

| **团队** | 个人探索 | 需要 | ⚠️ |

**创业可能性的突破口：**

如果要创业，不应该做"给个人开发者省 token 的工具"，而应该做：

> **"企业级 AI 代码生成的约束治理平台"**

场景：企业内部 200 个开发者都在用 AI 生成代码，但无法统一保证：
- 只使用公司批准的技术栈
- 遵循内部编码规范
- 不引入未授权的外部依赖
- 输出格式符合 CI/CD 要求

这个场景的 PCV 分数会高得多：企业有预算、有痛点、有合规压力。

## 十、总结与建议

### 作为开源实验项目：✅ 推荐推进

但需要做以下调整：

| # | 调整项 | 优先级 | 说明 |
|---|-------|-------|------|
| 1 | **将 NL→DSL 转换明确为本地/规则化方案** | 🔴 P0 | 解决核心矛盾 |
| 2 | **重新定义价值主张为"约束标准化/可复用/可审计"** | 🔴 P0 | "省 token" 的价值正在消失 |
| 3 | **增加 Prompt Caching 对比组** | 🟡 P1 | 确保 DSL 在缓存时代仍有优势 |
| 4 | **增加 .cursorrules 对比组** | 🟡 P1 | 与直接竞品对比 |
| 5 | **项目命名去"古文"化** | 🟡 P1 | 降低传播阻力 |
| 6 | **设计 .codespec 文件规范** | 🟢 P2 | 生态基础 |
| 7 | **做 VS Code 插件 MVP** | 🟢 P2 | 用户触达 |

### 作为创业项目：⚠️ 暂不建议

除非你能 pivot 到"企业级约束治理"赛道，并找到 3-5 个 design partner。

### 终极评判

> **CodePrompt-DSL 的本质价值不是"用古文省 token"，而是"给 LLM 交互建立工程约束的标准化层"。**
>
> 这就像 `.editorconfig` 之于编辑器、`.eslintrc` 之于代码风格——它的价值在于让约束变得显式、可复用、可共享、可审计。
>
> 如果沿着这个方向定位，项目的长期价值会远超"省 15% token"。
